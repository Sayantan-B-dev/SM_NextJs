# Client-side Rendering (CSR)

## Introduction

Client-Side Rendering (CSR) was the dominant pattern in early React applications, especially Single Page Applications (SPAs). In CSR, the server sends a minimal HTML shell, and JavaScript running in the browser generates the full UI.

## How CSR Works

### Traditional SPA Flow

```
Server                                Browser
  |                                     |
  |-- Sends index.html ---------------->|
  |   <div id="root"></div>            |
  |   <script src="bundle.js"></script> |
  |                                     |-- Load bundle.js
  |                                     |-- Parse & execute JS
  |                                     |-- Generate DOM
  |                                     |-- Inject into #root
  |                                     |-- UI becomes visible
```

### What the Browser Receives

```html
<!-- index.html from Create React App -->
<!DOCTYPE html>
<html lang="en">
  <head>
    <title>React App</title>
  </head>
  <body>
    <div id="root"></div>
    <script src="/static/js/bundle.js"></script>
  </body>
</html>
```

The `bundle.js` file contains:
- The React library
- All application components
- Routing logic
- State management

### What the Browser Renders

After JavaScript executes, the DOM is populated:

```html
<div id="root">
  <header>
    <img src="logo.png" alt="Logo" />
    <h1>Welcome to React</h1>
  </header>
  <main>
    <p>This content was generated by JavaScript.</p>
  </main>
</div>
```

## Verifying CSR

Open any CSR application and compare:

| Source View | DOM Inspector |
|---|---|
| `<div id="root"></div>` | Full rendered HTML with all elements |
| Bare minimum HTML | Complete UI structure |
| No meaningful content | All content visible |

## Drawbacks

### 1. SEO Problems

Search engine crawlers primarily read HTML content. With CSR, the initial HTML is essentially empty.

```
Crawler requests page
        |
        v
Receives: <div id="root"></div>
        |
        v
Cannot index any meaningful content
        |
        v
Poor search rankings
```

### 2. Poor User Experience

Users face a delay before seeing content:

```
Time --> |--- Download JS ---|--- Parse JS ---|-- Execute --|-- Render --|
         |                     |               |             |
         |      Blank page     |  Loading...   |   UI appears|
         |<-------- Time to First Meaningful Paint ---------->|
```

The user stares at a blank screen or loading spinner while:
- JavaScript downloads (potentially large bundles)
- JavaScript is parsed and compiled
- React generates the DOM

### 3. JavaScript Dependency

If JavaScript fails to load or is disabled, the application is completely unusable.

## When to Use CSR

| Use CSR When | Avoid CSR When |
|---|---|
| Internal dashboards | Public-facing content sites |
| Highly interactive apps | E-commerce / product pages |
| Apps behind authentication | SEO-critical pages |
| Tools and editors | Content-heavy sites |

## Visual Summary

```
CSR Architecture:

[Server]                          [Browser]
  |                                   |
  |--- index.html (bare) ----------->|
  |--- bundle.js (all logic) ------->|  Download
  |                                   |  Parse
  |                                   |  Execute
  |                                   |  Render
```

## Key Points

- Server sends minimal HTML with a JS bundle reference
- Browser downloads, parses, and executes JavaScript to generate UI
- View Source shows empty HTML; DOM inspector shows full content
- Two main drawbacks: poor SEO and slow initial content visibility
- Suitable for authenticated apps and dashboards
- Not ideal for SEO-critical public pages
